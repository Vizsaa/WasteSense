Part 1: USER ACCOUNT (Resident) -         BRUTAS
This module defines the functional requirements for Residents, the primary end-users who submit waste for collection and consume schedule information.
________________


1.1 Authentication & Access[a]
FR-UA-1.1: The system shall allow unregistered users to create a Resident account via a registration page.
FR-UA-1.2: The system shall require email, password, and profile details during registration.
FR-UA-1.3: The system shall allow Residents to log in using their registered email and password.
FR-UA-1.4: The system shall allow Residents to log out, terminating their session.
FR-UA-1.5: The system shall not provide automated password recovery; Residents must contact the Admin for password resets.
FR-UA-1.6: The system shall maintain user sessions for 24 hours unless manually logged out.
________________


1.2 Dashboard & Schedule Viewing[b]
FR-UA-1.7: The system shall display a Resident Dashboard immediately upon login.
FR-UA-1.8: The system shall display upcoming waste collection schedules specific to the Resident’s location/area.
FR-UA-1.9: The system shall display collection details including date, time, and waste category type.
FR-UA-1.10: The system shall not provide real-time vehicle tracking; schedules displayed shall be static and fixed.
________________


1.3 Waste Submission[c]
FR-UA-1.11: The system shall provide a "Submit Waste" feature accessible from the Resident Dashboard.
FR-UA-1.12: The system shall allow Residents to upload an image of the waste item(s).
FR-UA-1.13: The system shall enforce a maximum file size of 5MB for image uploads.
FR-UA-1.14: The system shall analyze the uploaded image using image recognition to suggest a waste category.
FR-UA-1.15: The system shall target a minimum accuracy of 92% for image recognition suggestions.
FR-UA-1.16: The system shall require the Resident to confirm or manually select the waste category before submission; the system shall not auto-finalize the category.
FR-UA-1.17: The system shall allow Residents to specify the collection location by entering a text address.
FR-UA-1.18: The system shall geocode the entered address to store precise coordinates for collector navigation.
FR-UA-1.19: The system shall submit the waste request with status "Pending" upon completion.
FR-UA-1.20: The system shall allow Residents to view a history of their past submissions and their statuses.
________________


1.4 Notifications
FR-UA-1.21: The system shall provide an in-app notification center visible when the Resident is logged in.
FR-UA-1.22: The system shall display notifications within the app when a submission status changes (e.g., Accepted, Collected).
FR-UA-1.23: The system shall display reminders for upcoming scheduled collections in the notification center.
________________


1.5 Feedback
FR-UA-1.24: The system shall provide a simple feedback form accessible from the Resident Dashboard.
FR-UA-1.25: The system shall allow Residents to submit general comments or feedback text to the Admin.
________________


Part 2: COLLECTOR ACCOUNT - AGUILA
This module defines the functional requirements for Collectors, the personnel responsible for accepting and completing waste collection tasks.
________________


2.1 Authentication & Access[d]
FR-CA-2.1: The system shall allow Collectors to log in using their assigned credentials.
FR-CA-2.2: The system shall allow Collectors to log out.
FR-CA-2.3: The system shall restrict access to the Collector Dashboard for non-Collector roles.
________________


2.2 Task Management & Assignment[e]
FR-CA-2.4: The system shall display a Collector Dashboard showing available pending waste submissions.
FR-CA-2.5: The system shall allow Collectors to view details of a pending submission (waste category, image, location).
FR-CA-2.6: The system shall allow Collectors to "Accept" a pending submission, thereby assigning the task to themselves.
FR-CA-2.7: The system shall prevent multiple collectors from accepting the same submission (first-to-accept logic).
FR-CA-2.8: The system shall move accepted submissions to a "My Tasks" list on the Collector Dashboard.
FR-CA-2.9: The system shall not automatically assign tasks; assignments occur only via self-assignment by the Collector.
________________


2.3 Collection Execution[f]
FR-CA-2.10: The system shall allow Collectors to view the geocoded location of the accepted submission on a map.
FR-CA-2.11: The system shall allow Collectors to update the status of a task to "Collected" upon completion.
FR-CA-2.12: The system shall prompt the Collector to confirm completion, updating the timestamp.
FR-CA-2.13: The system shall trigger a notification to the Resident once the status is updated to "Collected."
________________


Part 3: LGU ADMIN ACCOUNT - BARLISO[g]
This module defines the functional requirements for the Local Government Unit (LGU) Administrators who manage the system, users, schedules, and view analytics.
________________


3.1 Authentication & Access
FR-AA-3.1: The system shall create a default Admin account during database setup (Email: admin@wastesense.ph / Password: admin123).
FR-AA-3.2: The system shall restrict access to the Admin Dashboard for non-Admin roles.
FR-AA-3.3: The system shall allow Admins to log in and log out.
________________


3.2 User Management
FR-AA-3.4: The system shall allow Admin to view a list of all registered users.
FR-AA-3.5: The system shall allow Admin to create new user accounts (Resident, Collector, or Admin roles).
FR-AA-3.6: The system shall allow Admin to edit user profile details.
FR-AA-3.7: The system shall allow Admin to deactivate user accounts.
FR-AA-3.8: The system shall allow Admin to manually reset passwords for any user account.
________________


3.3 Configuration Management
FR-AA-3.9: The system shall allow Admin to define custom waste type categories (e.g., Biodegradable, Recyclable, Hazardous) via the dashboard.
FR-AA-3.10: The system shall make these custom categories available for selection during Resident submission.
FR-AA-3.11: The system shall allow Admin to manage Locations (Barangays/Streets) within the system.
FR-AA-3.12: The system shall allow the architecture to scale for multiple communities, even if initially deployed for a single community.
________________


3.4 Schedule Management
FR-AA-3.13: The system shall allow Admin to create collection schedules.
FR-AA-3.14: The system shall allow Admin to associate schedules with specific Locations and custom Waste Categories.
FR-AA-3.15: The system shall allow Admin to set collection day and time for schedules.
FR-AA-3.16: The system shall allow Admin to activate or deactivate schedules.
________________


3.5 Oversight & Analytics
FR-AA-3.17: The system shall display an Admin Dashboard with an overview of system statistics.
FR-AA-3.18: The system shall allow Admin to view all waste submissions and their current statuses.
FR-AA-3.19: The system shall display analytics including waste volume heat maps and temporal patterns.
FR-AA-3.20: The system shall display performance tracking metrics (e.g., submission counts, collection completion rates).
FR-AA-3.21: The system shall provide view-only access to analytics; export functionality is not required.
FR-AA-3.22: The system shall retain historical data indefinitely without automatic deletion.
________________


3.6 Feedback Management
FR-AA-3.23: The system shall allow Admin to view feedback submitted by Residents.
FR-AA-3.24: The system shall display feedback entries in a list format within the Admin Dashboard.
________________


Part 4: UI/UX ELEMENTS - BAYANI
This module defines the user interface and user experience requirements applicable to all roles and the system generally.
________________


4.1 Platform & Accessibility
FR-UX-4.1: The system shall be a web-based application accessible via standard web browsers.
FR-UX-4.2: The system shall not provide a native mobile application in the current scope.
FR-UX-4.3: The system shall implement responsive design to ensure accessibility on mobile browsers.
FR-UX-4.4: The system shall use English as the default and only language.
________________


4.2 Navigation & Layout
FR-UX-4.5: The system shall provide a Login Page as the default landing page for unauthenticated users.
FR-UX-4.6: The system shall provide a Registration Page for new users.
FR-UX-4.7: The system shall redirect users to a role-specific Dashboard upon login (Resident, Collector, or Admin Dashboard).
FR-UX-4.8: The system shall display a navigation menu appropriate to the user's role.
________________


4.3 Interactive Components
FR-UX-4.9: The system shall provide an image upload interface with file selection and preview capabilities.
FR-UX-4.10: The system shall provide interactive forms with validation feedback for required fields.
FR-UX-4.11: The system shall provide a map interface to visualize collection locations (static display based on geocoded addresses).
FR-UX-4.12: The system shall provide an in-app notification bell or list view accessible from the header/sidebar.
________________


4.4 Feedback & Status
FR-UX-4.13: The system shall display success/error messages after key actions (e.g., "Submission Successful", "Login Failed").
FR-UX-4.14: The system shall display status badges for waste submissions (e.g., "Pending", "Accepted", "Collected").
________________


4.5 Security & Session UI
FR-UX-4.15: The system shall protect pages based on authentication status; unauthenticated access redirects to Login.
FR-UX-4.16: The system shall persist user sessions for 24 hours, allowing users to remain logged in.
